# reverse-engineer

Explain this as if I'm going to build it myself from scratch. Start from the
problem it solves and the constraints that shaped it. Walk the design
decisions in the order a builder would face them: at each step, say what the
naive approach would be, why it fails, and what the real system does instead.
End with the two or three details that people who only read about it (rather
than build it) always get wrong. No history lesson, no feature tour — only
what I'd need to reconstruct it.
